メモリシステム
セッションをまたいで情報を永続化する仕組み。ユーザーの好み・プロジェクト文脈・フィードバックを記憶し、次回以降の会話品質を向上させる。

メモリの仕組み

Claude Code はプロジェクトごとに ~/.claude/projects/<project-hash>/memory/ ディレクトリにメモリファイルを保存する。
各メモリは個別の .md ファイルで、MEMORY.md がインデックスとして機能する。
~/.claude/projects/<project-hash>/memory/
├── MEMORY.md  # インデックス（毎セッション自動読み込み）
├── user_role.md  # ユーザー情報
├── feedback_testing.md  # フィードバック
└── project_auth.md  # プロジェクト情報

メモリの4つの型

user重要
ユーザーの役割・スキル・好みに関する情報。回答のレベル感やトーンを調整するために使われる。 
例: 「Goは10年経験あるがReactは初めて」
→ フロントエンドの説明をバックエンドの概念で例える
feedback重要
「こうして / こうしないで」というユーザーからの指示。修正だけでなく、うまくいったアプローチも記録される。 
例: 「テストでDBをモックしないで」
→ 理由: モックと本番の乖離でバグを見逃した経験
project
進行中の作業・目標・期限など、コードやgit履歴からは読み取れない文脈情報。 
例: 「木曜以降は非クリティカルなマージ禁止」
→ モバイルチームのリリースブランチカットのため
reference
外部システムの参照先。「どこを見ればいいか」を記録する。 
例: 「パイプラインのバグは Linear の INGEST プロジェクトで管理」
例: 「オンコール用ダッシュボードは grafana.internal/d/api-latency」

メモリファイルの書き方

各メモリファイルは YAML フロントマター + 本文の形式で記述する。 
---
name: テスト方針
description: 結合テストではDBをモックしない方針
type: feedback
---

結合テストでは必ず実DBに接続する。モックは使わない。

**Why:** 前四半期にモックテストがパスしたが、
本番マイグレーションで失敗した事故があった。

**How to apply:** テストファイルを書く際、
DB接続部分にモックを使おうとしたら止める。
MEMORY.md はインデックスファイル。各エントリは1行150文字以内で、200行を超えると切り捨てられる。メモリ本文は書かず、ファイルへのリンクだけを記載する。

メモリの操作

記憶させる
「〜を覚えておいて」と自然言語で伝えるだけでよい。Claude が適切な型を判断してファイルを作成する。 
「テストは必ず実DBで動かす方針を覚えておいて」
「私はデータサイエンティストです、覚えておいて」
忘れさせる
「〜を忘れて」と伝えると、該当するメモリファイルを削除する。 
「テスト方針のメモリを削除して」
自動記録
明示的に頼まなくても、Claude は会話中に重要な情報を検知して自動的にメモリに保存することがある（ユーザーの修正、アプローチの確認など）。

メモリに保存しないもの

以下はメモリに保存しない。コードやgitから取得できる情報は常にそちらが正となる。

× コードパターン・アーキテクチャ・ファイル構造（コードから読める）
× git 履歴・変更者情報（git log / git blame で取得）
× デバッグの解決策（修正はコードに、文脈はコミットメッセージにある）
× CLAUDE.md に既に書かれている情報
× 現在の会話でしか使わない一時的な情報（タスクを使う）
