エージェントチーム
カスタムサブエージェントの定義と、複数エージェントによるチーム協調作業。

カスタムサブエージェント定義

.claude/agents/ にMarkdownファイルを置くことで、専門的な役割を持つサブエージェントを定義できる。
YAML frontmatter で設定を記述し、本文がそのエージェントのシステムプロンプトになる。
ファイル配置
.claude/agents/reviewer.md    ← プロジェクト単位
~/.claude/agents/reviewer.md ← 全プロジェクト共通
基本的な定義ファイル
---
name: code-reviewer
description: コードレビューを実行する。コード変更後に proactively 使用する。
tools: [Read, Glob, Grep, Bash]
model: sonnet
---

あなたはコードレビュアーです。
コード品質・セキュリティ・ベストプラクティスの観点で
具体的かつ実用的なフィードバックを提供してください。

frontmatter フィールド一覧

name必須
一意な識別子。小文字とハイフンのみ使用可能。
description必須
Claude がどんなタスクでこのエージェントに委譲すべきか判断する際の説明文。use proactively を含めると自動委譲が促進される。
tools
使用可能なツールの許可リスト。省略時は全ツール継承。Agent(worker, researcher) のようにサブエージェント制限も可能。
disallowedTools
拒否するツールのリスト。tools より先に適用される。
model
sonnet / opus / haiku / フルモデルID / inherit（デフォルト）
permissionMode
default / acceptEdits / auto / dontAsk / bypassPermissions / plan
maxTurns
エージェントが停止するまでの最大ターン数。暴走防止に有効。
memory
永続メモリのスコープ。user（全プロジェクト）/ project（VCS共有）/ local（ローカルのみ）
mcpServers
このエージェントが利用する MCP サーバー。既存サーバー名の参照またはインライン定義が可能。
hooks
このエージェントにスコープされたライフサイクルフック。PreToolUse / PostToolUse / Stop
skills
起動時にコンテキストに読み込むスキル。利用可能にするだけでなく全文が注入される。
isolation
worktree を設定すると、独立した git worktree で実行。変更がなければ自動クリーンアップ。
background
true で常にバックグラウンドタスクとして実行。
effort
low / medium / high / max（Opus のみ）
color
表示色。red / blue / green / yellow / purple / orange / pink / cyan
initialPrompt
--agent でセッション起動時に自動送信される最初のプロンプト。

エージェントの呼び出し方法

自動委譲
Claude が description を読み、タスクに合致するエージェントに自動で委譲する。
自然言語で指示
「code-reviewer エージェントで最近の変更をレビューして」
@メンション確実
@"code-reviewer (agent)" auth モジュールを確認して
確実にそのエージェントが実行される。
セッション全体で使用
claude --agent code-reviewer
セッション全体がそのエージェントとして動作する。
/agents コマンド
対話的にエージェントの管理・作成・編集ができる。

定義の優先順位

同名エージェントが複数ある場合、上位が優先
1. Managed settings（組織配布）
2. --agents CLI フラグ（セッション限定）
3. .claude/agents/（プロジェクト）
4. ~/.claude/agents/（ユーザー全体）
5. プラグインの agents/

実践例: カスタムサブエージェント

テストランナー
---
name: test-runner
description: テストスイートを実行して失敗を報告する
tools: [Bash, Read]
model: haiku
maxTurns: 20
---

npm test を実行し、失敗があれば
ファイルパスと行番号を含めて報告してください。
DB クエリ（フック付き）
---
name: db-reader
description: 読み取り専用のデータベースクエリを実行する
tools: [Bash]
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-readonly.sh"
---

読み取り専用クエリのみ実行してください。
書き込み操作は絶対に行わないこと。
ブラウザテスター
---
name: browser-tester
description: Playwright でブラウザテストを実行する
mcpServers:
  - playwright:
      type: stdio
      command: npx
      args: ["-y", "@playwright/mcp@latest"]
---

Playwright ツールでページの操作・スクショ・
検証を行ってください。

エージェントチーム（実験的機能）

エージェントチームは、複数の独立した Claude Code セッションが共有タスクリストとメッセージングで協調作業する仕組み。
サブエージェントとは異なり、チームメイト同士が直接コミュニケーションできる。
実験的機能 — デフォルトでは無効。有効化が必要。 
// settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}

サブエージェント vs エージェントチーム

サブエージェント
結果を返すだけの一方通行。
• 親エージェントにのみ報告
• 個別のコンテキストウィンドウ
• 結果は要約されて返る
• トークンコストが低い
• 向いている: 結果だけが必要な集中タスク
エージェントチーム
チームメイト同士が議論・協調できる。
• メンバー間で直接メッセージ送受信
• 共有タスクリストで自己調整
• 完全に独立したセッション
• トークンコストが高い
• 向いている: 議論・協調が必要な複雑な作業

チームの構成

チームアーキテクチャ
Team Lead（メインセッション）
├── Teammate A: UX 設計を担当
├── Teammate B: 技術アーキテクチャを担当
├── Teammate C: セキュリティレビューを担当
├── 共有タスクリスト（pending → in progress → completed）
└── メッセージング（個別送信 / ブロードキャスト）
Team Lead
メインの Claude Code セッション。チームの作成・メンバー召喚・全体調整を行う。
Teammate
独立した Claude Code インスタンス。割り当てられたタスクを実行し、他メンバーとメッセージで連携する。
共有タスクリスト
チーム全体で共有。3つのステータス: pending / in progress / completed。タスク間の依存関係も設定可能。Ctrl+T で表示切替。
メッセージング
message: 特定メンバーに送信。broadcast: 全メンバーに送信（コスト注意）。リードへのアイドル通知は自動。

チームの起動方法

自然言語でタスクとチーム構成を伝えるだけで、Claude がチームを構成・起動する。 
# 例: 並列コードレビュー
PR #142 のレビューチームを作って。3人のレビュアーを召喚:
- セキュリティの観点で確認する人
- パフォーマンスへの影響を確認する人
- テストカバレッジを検証する人
それぞれレビューして結果を報告させて。
# 例: 競合仮説によるデバッグ
アプリが1メッセージ後に終了するバグがある。
5人のチームメイトを召喚して異なる仮説を調査させて。
お互いの理論を反証し合う科学的議論の形式で進めて。

サブエージェント定義をチームメイトとして使用

.claude/agents/ で定義したエージェントをチームメイトとして召喚できる。
tools と model が継承され、本文はシステムプロンプトに追記される。 
# 自然言語で指示
security-reviewer エージェントタイプを使って
チームメイトを召喚し、auth モジュールを監査して。
制限: チームメイトとして使用する場合、skills と mcpServers フィールドは適用されない。MCP サーバーはプロジェクト/ユーザー設定から読み込まれる。

表示モード

in-processデフォルト
全メンバーがメインターミナル内で実行。Shift+Down でメンバーを切り替え。
tmux / iTerm2
各メンバーが独立したペインで実行。tmux または iTerm2 が必要。 
# ~/.claude.json で設定
{ "teammateMode": "tmux" }

# またはセッション単位
claude --teammate-mode tmux

チーム用フック

TeammateIdle
チームメイトがアイドルになろうとした時に発火。終了コード 2 でフィードバックを送り作業を継続させる。
TaskCreated
タスク作成時に発火。終了コード 2 で作成を阻止。
TaskCompleted
タスク完了時に発火。終了コード 2 で完了を阻止。

ベストプラクティス

チームサイズ
3〜5人で開始。1人あたり5〜6タスクが目安。大きすぎるチームはコスト増とコミュニケーション負荷が増す。
十分なコンテキスト
チームメイトはリードの会話履歴を継承しない。タスク固有の詳細を召喚プロンプトに含めること。
ファイル競合の回避
2人が同じファイルを編集すると上書きが発生する。各メンバーが異なるファイルを担当するように設計する。
まずは読み取りから
初めて使う場合は、コード変更ではなくレビュー・調査タスクから始めるのが安全。
リードの待機
リードが自分で実装を始めてしまう場合は「チームメイトの完了を待って」と指示する。

現在の制限事項

• /resume / /rewind でチームメイトは復元されない
• タスクステータスの更新が遅延することがある
• シャットダウンに時間がかかる場合がある（現在のリクエスト完了待ち）
• 1セッションにつき1チームのみ
• チームのネストは不可（チームメイトが独自チームを作れない）
• 分割ペインは tmux / iTerm2 が必要（VS Code / Windows Terminal 非対応）
