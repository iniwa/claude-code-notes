併用ワークフロー
Codex と Claude Code を、曖昧さ・設計リスク・実行量に応じて切り替える。
判断基準

このプロジェクトでは「Codex で設計して Claude Code で実装」と固定しない。Codex は実装まで進められるため、役割ではなく作業の性質で切り替える。

  • Codex 優先: 要求が曖昧、設計判断が絡む、差分の意味をレビューしたい、記録まで一気通貫で閉じたい。
  • Claude Code 優先: handoff が明確、編集量が多い、CLI/Hooks/サブエージェント/CI など Claude Code 固有の実行環境を使いたい。
  • 併用: Codex が方針と制約を整理し、Claude Code が実装し、Codex が差分と設計記録を確認する。
1
Codex で変更の性質を判定する
まず、今回の作業が新規機能、既存機能の調整、単純な修正、ドキュメント更新、自動化作業のどれかを切り分ける。
同時に、設計判断が必要か、実装範囲が明確か、どちらの実行環境が向いているかを決める。 
この変更は設計判断を含みますか？
実装範囲は明確ですか？
Codex がこのまま実装まで進めるべきですか？
それとも Claude Code に handoff した方が速いですか？
2
設計リスクがある場合は Codex で整理する
責務境界、データ構造、公開 API、永続化、認証、デプロイ、将来の拡張に影響する場合は、先に Codex で設計意図を固める。
この時点では、毎回 AGENTS.md を更新しない。今後も守るべき判断が増えたときだけ記録する。 
Context:
- 何が問題だったか

Decision:
- 何を採用したか

Reason:
- なぜその判断にしたか

Do Not Change Casually:
- 何を不用意に変えてはいけないか
3
Codex で進めるか Claude Code に渡すか決める
小さく閉じる変更、レビューと実装を同じ文脈で進めたい変更、設計記録まで一気通貫で扱う変更は Codex がそのまま実装する。
一方、明確な編集指示に落ちた大きな変更、反復的な修正、Claude Code の Hooks やサブエージェントを使う作業は Claude Code に渡す。 
Codex で続行:
- 曖昧さが残っている
- 設計レビューしながら編集したい
- 変更範囲が小さく、検証までこの場で閉じられる

Claude Code に handoff:
- 目的、編集対象、制約、検証方法が明確
- 編集量が多い
- Claude Code の自動化や並列作業が効く
4
Claude Code に渡す場合は handoff を作る
Claude Code には「良い感じに実装して」ではなく、目的、編集対象、制約、非目標、検証方法を渡す。
とくに既存機能の調整では、許可する変更と許可しない変更を明確にする。 
## Goal
○○を実装する。

## Design Intent
この変更で守るべき設計意図を書く。

## Files To Edit
- path/to/file

## Constraints
- 既存 API を維持する
- UI 層にビジネスロジックを置かない

## Non Goals
- DB スキーマは変更しない

## Verification
- npm test
- npm run build
5
実装結果を構造化して受け取る
Codex で実装した場合も Claude Code に渡した場合も、最後は変更内容、検証結果、設計制約との関係を残す。
実装者名よりも、差分がどの判断に基づいているかを確認しやすくする。 
## Changed Files
- path/to/file

## Summary
- What changed?

## Design Constraints
- Which constraints were preserved?

## Verification
- Commands run
- Results

## Open Questions
- Anything to record in AGENTS.md or docs?
6
Codex で設計整合性レビューを行う
実装差分を AGENTS.md の設計思想と照合する。
余計な責務移動、一時対応の恒久化、handoff 外の設計変更がないかを確認する。 
以下の実装差分を、AGENTS.md の設計思想と照らしてレビューしてください。
特に、途中で設計思想の剥離が起きていないか確認してください。
7
必要なら AGENTS.md を更新する
実装中に新しい制約や判断が発生した場合だけ、Codex 側で AGENTS.md や docs/ を更新する。
単純なバグ修正、文言修正、CSS 微調整、既存方針に沿った小変更では更新しなくてよい。
