併用ワークフロー
Codex で設計を練り、Claude Code で実装し、Codex で設計整合性を確認する。
1
Codex で要件と変更種別を整理する
まず、今回の作業が新規機能なのか、既存機能の調整なのか、単純な修正なのかを切り分ける。
新規機能では「設計に新しい概念を追加してよいか」、既存機能では「設計変更が必要か、実装だけ直せばよいか」を確認する。 
これは AGENTS.md に残すべき設計判断を含む変更ですか？
含むなら、実装前に handoff と decision log を作ってください。
含まないなら、Claude Code に渡す最小の実装指示だけ作ってください。
2
AGENTS.md に設計思想を残す
Codex 側では、なぜその設計にしたのか、何を変えてはいけないのか、将来どこを見直してよいのかを記録する。
毎回更新する必要はない。責務境界、採用しなかった案、今後も守るべき制約が増えたときに更新する。 
Context:
- 何が問題だったか

Decision:
- 何を採用したか

Reason:
- なぜその判断にしたか

Do Not Change Casually:
- 何を不用意に変えてはいけないか
3
Claude Code 用 handoff を作る
Claude Code には「良い感じに実装して」ではなく、目的、編集対象、制約、非目標、検証方法を渡す。
とくに既存機能の調整では、許可する変更と許可しない変更を明確にする。 
## Goal
○○を実装する。

## Design Intent
この変更で守るべき設計意図を書く。

## Files To Edit
- path/to/file

## Constraints
- 既存 API を維持する
- UI 層にビジネスロジックを置かない

## Non Goals
- DB スキーマは変更しない

## Verification
- npm test
- npm run build
4
Claude Code で実装する
Claude Code は CLAUDE.md と handoff に従って実装する。
CLAUDE.md には、AGENTS.md と docs/ の設計判断を制約として扱うように書いておく。 
AGENTS.md の設計方針と以下の handoff に従って実装してください。
設計制約から外れる変更が必要な場合は、実装前に止まって確認してください。
5
実装結果を構造化して受け取る
Claude Code には、変更内容だけでなく、設計制約をどう守ったか、検証結果、未確認事項を返してもらう。
この形式にしておくと、Codex での設計整合性レビューがしやすい。 
## Changed Files
- path/to/file

## Summary
- What changed?

## Design Constraints
- Which constraints were preserved?

## Verification
- Commands run
- Results

## Open Questions
- Anything Codex should review or record?
6
Codex で設計整合性レビューを行う
実装差分を AGENTS.md の設計思想と照合する。
余計な責務移動、一時対応の恒久化、Claude Code の独自判断による設計変更がないかを確認する。 
以下の実装差分を、AGENTS.md の設計思想と照らしてレビューしてください。
特に、途中で設計思想の剥離が起きていないか確認してください。
7
必要なら AGENTS.md を更新する
実装中に新しい制約や判断が発生した場合だけ、Codex 側で AGENTS.md や docs/ を更新する。
単純なバグ修正、文言修正、CSS 微調整、既存方針に沿った小変更では更新しなくてよい。
