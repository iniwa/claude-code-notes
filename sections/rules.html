プロジェクトルール
Codex と Claude Code を併用するプロジェクトで、最初に決めておくと作業が安定するルールとフォルダ構成。

最初に決めること

プロジェクトの形
  • アプリ種別、実行方法、デプロイ先を明文化する。
  • ビルド有無、主要言語、依存管理方法を決める。
  • entry point と主要ディレクトリを一覧化する。
AI の役割分担
  • Codex は意図整理、設計判断、handoff、レビューを担当する。
  • Claude Code は明確な指示に基づく実装、検証、報告を担当する。
  • 曖昧な要求は実装に渡す前に Codex で具体化する。
公開・安全ルール
  • API キー、認証情報、個人トークンは置かない。
  • private な運用情報はテンプレート化しない。
  • 個人環境の値は placeholder にする。

推奨フォルダ構成

.
|-- AGENTS.md Codex 向け。意図、役割分担、設計原則、handoff ルール
|-- CLAUDE.md Claude Code 向け。実装ルール、検証、報告形式
|-- src/ アプリケーション本体。プロジェクトに合わせて命名
|-- tests/ 自動テスト。単体、統合、回帰テスト
|-- docs/ 作業判断、handoff、設計記録、運用手順
| |-- handoffs/ Codex から Claude Code への実装指示
| |-- decisions/ 長めの設計判断、背景、却下案
| `-- runbooks/ 繰り返し使う運用手順。必要になってから作る
|-- templates/ 再利用するプロンプト、設定例、文書雛形
|-- tools/ 補助スクリプト、メンテナンスツール
`-- README.md 人間向けの概要、起動方法、基本コマンド

docs 配下の使い分け

docs/handoffs/
Codex が Claude Code に実装を渡すための scoped handoff を置く。1 ファイル 1 タスクにし、Goal / Files / Constraints / Verification を必ず含める。
docs/decisions/
AGENTS.md に入れるには長すぎる背景、判断理由、却下案を置く。未来の作業判断で再利用できる内容にする。
docs/runbooks/
デプロイ、復旧、リリース、定期作業など、手順として繰り返し使う文書を置く。必要になるまで作らない。

AGENTS.md に書くこと

  • このプロジェクトの目的、技術構成、主要ディレクトリ。
  • Codex と Claude Code の責任範囲。
  • 設計原則、禁止事項、触ってはいけないファイル。
  • handoff の作り方、レビュー観点、検証方針。
  • 将来のセッションにも効く短い永続ルール。

CLAUDE.md に書くこと

  • Claude Code が作業開始時に読むファイル。
  • 編集してよい範囲、止まって確認すべき条件。
  • テスト、lint、型チェック、起動確認などの検証コマンド。
  • 最後に返す報告形式。changed files / summary / verification / blocked checks を含める。
  • ローカル設定や secret を触らないルール。

handoff の最小単位

  • Goal: 何を達成するか。
  • Background: なぜその変更をするか。
  • Files To Inspect: 実装前に読むファイル。
  • Files To Edit: 編集してよいファイル。
  • Constraints: 守るべき制約。
  • Non Goals: 今回やらないこと。
  • Verification: 実行する確認。
  • Expected Report: Claude Code が返す報告形式。

作業中の判断ルール

  • 要求が曖昧なら、実装せず Codex で目的と制約を固める。
  • 編集量が多く、範囲が明確なら Claude Code に handoff する。
  • 設計変更や責任境界の変更は、実装前に Codex で判断する。
  • 単発の作業ログは残さず、再利用する判断だけ文書化する。
  • 実装後は差分が handoff とプロジェクトルールに沿っているかレビューする。
