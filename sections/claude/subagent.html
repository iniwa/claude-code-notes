サブエージェント実践パターン
サブエージェントの基礎知識と、ユーザーが介入できるポイント。

サブエージェントとは

サブエージェントとは、Claude Code が内部で起動する「別プロセスの Claude」。
通常は1つの会話でメインの Claude が作業するが、複雑なタスクでは別の Claude を立ち上げて作業を分担させることができる。
メインの Claude（ユーザーと会話中）
├── サブエージェント A: フロントエンドのコードを調査
├── サブエージェント B: バックエンドの構造を調査
└── 結果を受け取って統合 → ユーザーに回答
基本的にユーザーが設定・操作する必要はない。
サブエージェントの起動タイミング・種類・モデル選択はすべて Claude が自律的に判断する。CLI フラグや設定ファイルでサブエージェントを直接制御する手段は存在しない。
ユーザーは「サブエージェントが何をしているか」を知っておくだけで十分。
独立したコンテキスト
サブエージェントはメインの会話とは別のコンテキストで動く。メインの会話履歴を圧迫せず、大量の調査結果も要約して返してくれる。
並列実行
複数のサブエージェントを同時に走らせて待ち時間を短縮できる。独立した調査タスクを並列化するのが典型的な使い方。
自律的に使われる
ユーザーが直接操作するものではなく、Claude が必要と判断したときに内部で自動起動する。ユーザーから見ると「Claude が勝手にやってくれている」もの。

サブエージェントの種類（参考）

Explore
コードベースの探索に特化した高速エージェント。ファイルパターン検索、キーワード検索、アーキテクチャ理解に使われる。
Plan
設計・実装計画の策定に特化。重要ファイルの特定やアーキテクチャのトレードオフ検討を行う。コード編集はできない（読み取り専用）。
general-purpose
汎用エージェント。複雑なマルチステップタスクを自律的に処理する。全ツールにアクセス可能。
どの種類を使うかは Claude が判断する。ユーザーが指定する必要はない。

ユーザーが介入できるポイント

自然言語での指示唯一の方法
サブエージェントの挙動に影響を与える唯一の手段は、自然言語で Claude に伝えること。 
# 会話中に伝える
「単純な検索にはコストの低いモデルを使って」
「並列で調査して」
「worktree を使って安全に実験して」
CLAUDE.md に方針を記載
毎回伝えるのが面倒なら、CLAUDE.md に方針を書いておく。すべてのセッションで自動的に読み込まれる。 
# CLAUDE.md に記載する例
## Tooling
- サブエージェントの単純な検索タスクには
  コストの低いモデルを優先すること
SubagentStop フック
サブエージェントが完了した時に発火するフック。完了通知を送ったり、追加の処理を自動実行したりできる。唯一の設定ファイルベースの介入ポイント。 
"hooks": {
  "SubagentStop": [{
    "hooks": [{
      "type": "command",
      "command": "echo 'サブエージェント完了'"
    }]
  }]
}

知っておくと便利な動作

Worktree 分離
Claude がサブエージェントを一時的な git worktree で実行することがある。メインのリポジトリに影響なく安全に実験的な変更を試みる仕組み。変更がなければ自動クリーンアップされる。
バックグラウンド実行
Claude がサブエージェントをバックグラウンドで走らせ、メインの作業と並行して処理することがある。完了すると結果が自動的にメインの会話に返される。

カスタムスキル（旧カスタムコマンド）

頻繁に使うエージェントパターンは スキル として定義しておくと、スラッシュコマンドで呼び出せる。 
# .claude/skills/review/SKILL.md に定義
---
description: コードレビューを実行する
---
以下のルールに基づいてコードレビューしてください:
- セキュリティ: OWASP Top 10 をチェック
- パフォーマンス: N+1 クエリ, 不要な再レンダリング
- 可読性: 命名、関数の長さ
/review で呼び出せるようになる。個人用は ~/.claude/skills/ に置くと全プロジェクトで使える。
